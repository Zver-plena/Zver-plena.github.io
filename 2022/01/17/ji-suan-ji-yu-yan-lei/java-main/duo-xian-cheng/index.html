多线程

用时:2021.12.1-2021.12.11

基本概念:程序,进程,线程

  • 程序(program) : 为完成特定任务,用某种语言编写的一组指令的集合.即一段静态的代码,静态对象.

  • 进程(process) : 是程序的一次执行过程.或是正在运行的一个程序,是一个动态的过程,有他自身的产生,存在和消亡的过程—–生命周期

    • 运行中的程序

    • 程序是静态的,进程是动态的

    • 进程作为资源分配的单位,系统在运行时会为每个进程分配不同的内存区域.

  • 线程(thread) : 进程可进一步细分划分为线程,是一个程序内部的一条执行路径.

    • 若一个进程同一时间并行执行多个线程,就是支持多线程的

    • 线程作为调度和执行的单位,每个线程具有独立的运行栈和程序计数器(pc),线程切换的开销小

    • 一个进程中的多个线程共享相同的内存单元/内存地址空间->他们从同一堆中分配对象,可以访问相同的变量和对象,这就使得线程间的通信变得更加简便和高效,但多个线程操作共享的系统资源可能带来安全隐患

  • 单核CPU与多核CPU

    • 单核CPU : 其实是一个假的多线程,在一个时间单元内只能执行一个线程的任务.只是因为CPU的时间单元很小,所以感觉不出来.
    • 多核CPU:更好的发挥多线程的效率

  • 并行 : 多个CPU同时执行多个不同的任务

  • 并发 : 一个CPU(采用时间片)同时执行多个任务

    使用多线程的优点

  • 以单核CPU为例子,只使用单个线程先后完成多个任务,肯定比用多个线程来完成的时间更加段短,既然如此为什么要使用多线程?

  • 多线程的优点

      1. 提高应用程序的响应,对图形化界面有意义,增强用户体验
      1. 提高计算机CPU的利用率
      1. 改善程序结构,将长而复杂的进程分为多个线程,独立运行,利于修改

        何时使用多线程

  • 程序需要执行多个任务

  • 程序需要实现一些需要等待的任务,如用户输入,文件读写操作,网络操作,搜索等

  • 需要一些后台运行的程序

多线程的创建

继承Thread类

线程的创建流程

  1. 创建一个继承Thread类的子类.
  2. 重写Thread类的run()–这个线程需要执行的操作申明在run里面
  3. 创建Thread类的子类的对象.
  4. 用对象调用Thread的start().
    1. 主线程启动新的线程
    2. 调用这个线程的run()方法

问题与解决

  1. 不能让已经start()的对象再去start(),会抛出IllegaleThreadException异常
  2. 创造多个线程,就要创造多个对象去调用一次start().
  3. 不同的方法就创造不同的类继承Thread类.
  4. 只用一次的可以用Thread类的匿名类写法

Thread的常用方法

  1. start(): 创建新的线程,调用线程的run()
  2. run():通常需要重写Thread类中的run方法,将创建该线程所需要完成的操作写在run()方法里面
  3. currentThread():获取当前进程
  4. getName():获取当前进程的名字
  5. setName():设置当前进程的名字
  6. yield():释放当前CPU的执行权 7. join():当线程a调用线程b的join()方法,那么a就会进入堵塞状态,直到b执行完,
    1. 低优先级的线程可以获得执行
  7. sleep(long millitime):让当前进程睡眠指定的millitime毫秒,在指定的时间内为阻塞状态
    1. 在线程中需要try-catch抛异常

实现Runnable接口

线程创建流程

  1. 创建一个实现Runnable接口的类
  2. 实现类去实现Runnable接口中的方法:run()
  3. 创建实现类的对象
  4. 将实现类的对象作为参数传递到Thread的构造器中,创造Thread的对象
  5. 通过Thread对象来调用start()

比较两种创建方法

优先选择实现Runnable接口

  1. 原因
    1. 实现的方法没有类的单继承的局限性
    2. 实现的方式更加适合处理多个线程享有共同资源的情况

两者的联系与相同点

  1. 联系
    1. 不管是继承Thread类还是实现Runnable接口,本质上都是在重写Runnable接口的run()方法,因为Thread类的run()方法也是重写了Runnable接口的run()方法
  2. 相同点
    1. 两种创建方法都需要重写run()方法,将线程需要执行的操作写在run()方法里面

多线程的安全问题

存在的安全问题

  1. 多个线程执行的不确定性引起结果的不稳定
  2. 多个线程对资源的共享,造成操作的不完整性,会破坏数据
  3. 对资源的共享才会带来多线程的安全问题

问题根源

当某个线程在操作共享资源时,尚未操作完成,其他线程也参与进来

解决方案

由此引入线程的同步

当一个线程在操作 共享数据的时候,其他线程不能参与进来,知道该线程操作完成,其他线程才能够开始操作共享数据,这种情况即使该线程出现阻塞也不能被改变

Java中通过同步机制来解决线程的安全问题

synchronized()方法同步代码块

Runnable接口
public void run(){
  
    synchronized(同步监视器){
       //需要被同步的代码(多个线程同步操作的数据)    
    } 

}

  • 同步监视器必须只为一个对象

  • 在Runnable接口中,因为对象只有一个所以同步监视器可以用this

继承Thread类
public void run(){
    
    synchronized(同步监视器){
       //需要被同步的代码(多个线程同步操作的数据)
    }
    
}
  • 在继承Thread类中,因为每次都会new新的对象所以同步监视器不可以用this
  • 可以用 Class.class来充当对象
同步代码块总结
  • 同步监视器:俗称锁,任何一个类的对象都可以充当一个锁
  • 要求:多个线程必须共用同一把锁

synchronized()同步方法

如果操作共享数据的操作在同一个方法里面,那么可以用syncchronized()来创造一个方法

实现Runnable接口

           show();
       }
   }

   }

同步方法总结
  • synchronized()实现同步方法默认的锁为this,这在实现Runnable接口时可以直接不写,当如果是继承Thread类则必须此方法为静态方法:静态方法的锁是当前类本身,也就是Class.class

Lock锁—-JDK5.0新增

Lock
  • java.util.concurrent.locks.Lock接口是控制多个线程对共享资源进行访问的工具
  • ReentrantLock类实现了Lock,他拥有与synchroized相同的并发性和内存语义,在现实线程安全的控制中,比较常用的是ReentrantLock,可以显式加锁,释放锁
  • 使用ReentrantLock:import java.util.concurrent.locks.ReentrantLock;引入这个包
Lock操作
ReentrantLock lock=new ReentrantLock();
public void run() {
        lock.lock();// lock.lock 必须紧跟 try
        try {
         	//同步方法   
        }finally{// 并且 lock.unlock 要在 finally 的第一行
            lock.unlock();
        }
    }
synchornized与Lock的异同
  1. 相同:
    1. 都是解决线程安全问题
  2. 不同:
    1. synchornized执行完同步代码块或者同步方法,自动释放同步监视器
    2. Lock同步监视器需要手动调用解锁方法Lock.nulock()
    3. Lock是显式锁(手动开关锁),synchronized是隐式锁,出了作用域自动释放
    4. Lock只有代码块锁,synchronized有代码块锁和方法锁
    5. 使用Lock锁,JVM花费更少的事件来调度线程,性能更好,并且具有更好的扩展性
优先使用顺序

Lock -> 同步代码块 -> 同步方法

总结

线程同步解决了线程的安全问题,但是也降低了线程的效率

单例模式:饿汉式的线程安全写法

class Person {
    private static Person instance=null;
    public static Person getInstance(){
        if(instance==null){
            synchronized(Person.class){
                instance=new Person();
            }
        }
        return instance;
    }
}

死锁问题

死锁的理解
  1. 不同的线程分别占用对方需要的资源不放,都在等待对方先放弃自己需要的同步资源,就形成了线程的死锁
  2. 说明
  3. 出现死锁后,不会出现异常,不会出现提示,只是所有的线程处于阻塞状态,无法继续.
  4. 使用同步时要避免死锁问题.
  5. 锁相当于一种资源,以这个资源的占有权来实现同步,一个线程同步需要这个锁,那么这个锁的资源将被该线程占有,其他线程无法使用.而这里的锁是以对象来充
解决方法
  1. 专门的算法,原则;
  2. 尽量减少同步资源的定义;
  3. 尽量避免嵌套同步;

线程的通信

通信方法

  1. wait()

    一旦执行这个方法,当前线程就会进入堵塞状态,并释放同步监视器

  2. notify()

    一旦执行这个方法,就会唤醒被wait的一个进程,如果有多个进程被wait(),就会唤醒优先级最高的一个

  3. notifyAll()

    一旦执行这个方法,就会唤醒所有被wait的进程.

这些方法都是在java.lang.Objects包中:确保任何一个类的对象充当同步监视器的时候都能够调用通信方法

注意事项

  1. wait(),notify(),notifyAll()三个方法必须使用在同步代码块或者同步方法中.

  2. wait(),notify(),notifyAll()三个方法的调用者必须是同步代码块或者同步方法中的同步监视器.

wait()和sleep()的异同

  1. 异
    1. 两个方法的声明位置不一样;Thread类中声明sleep(),Object类中声明wait().
    2. 调用的场景不同;sleep()可以在任何场景中调用,而wait()只能在同步代码或者同步方法中使用.
    3. 关于是否释放同步监视器;如果两个方法都使用在同步代码块或者同步方法中,sleep()不会释放同步监视器,wait()会释放同步监视器
  2. 同
    1. 一旦执行,就会让线程进入阻塞状态

生产者/消费者问题—线程通信的经典例题

生产者（Producer）将产品交给店员（Clerk），而消费者（Customer）从店员处取走产品，I店员一次只能持有固定数量的产品（比如：20），如果生产者试图生产更多的产品，店员会叫生产者停一下，如果店中有空位放产品了再通知生产者继续生产；如果店中没有产品了，店员会告诉消费者等一下，如果店中有产品了再通知消费者来取走产品。

这里可能出现两个问题：

  1. 生产者比消费者快时，消费者会漏掉一些数据没有取到。

  2. 消费者比生产者快时，消费者会取相同的数据。

对多线程问题的分析:

  1. 是否是多线程?

    2. 消费者线程—Customer

  2. 是否有共享数据

    1. 有
    2. 店员(产品)—Cleark

  3. 如何解决线程的安全问题

    1. 同步机制—三种方法

  4. 是否涉及线程的通信

class Clerk {
    private int productCount = 0;
    private final int productMaxcount = 20;

    public synchronized void produceProduct() {
        if (productCount < productMaxcount) {

            productCount++;
            System.out.println(Thread.currentThread().getName() + "：开始生产第" + productCount + "个产品");
            notify();
        } else {
            //等待
            try {
                wait();
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }

    public synchronized void consumeProduct() {
        if (productCount > 0) {
            System.out.println(Thread.currentThread().getName() + "：开始消费第" + productCount + "个产品");
            productCount--;
            notify();
        } else {
            //等待
            try {
                wait();
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }

}

class Customer extends Thread {
    private Clerk clerk;

    public Customer(Clerk clerk) {
        this.clerk = clerk;
    }

    @Override
    public void run() {
        while (true) {
            try {
                Thread.sleep(100);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }

            clerk.consumeProduct();
        }
    }
}

        }
    }
}

JDK5.0新增线程创建方式

实现Callable接口

线程创建流程

  1. 创建一个实现Callable接口的实现类
  2. 实现call()方法,线程任务声明在call()方法里面
  3. 创建一个实现Callable接口的实现类的对象
  4. 将Callable接口的实现类的对象作为参数传递到FutureTask构造器中,创建FutureTask的对象
  5. 将FutureTask类的对象作为参数传递到Thread类的构造器中,并且调用start()方法.
  6. (可有可无)获取Callable中call()方法的返回值:FutureTask.get()

代码实现

package com.lgx.ThreadNew;

import java.util.concurrent.Callable;
import java.util.concurrent.ExecutionException;
import java.util.concurrent.FutureTask;

/**
 * @ClassName Test
 * @Author mrl
 * @Date 2021/12/10 22:48
 * @Version 1.6
 * @Description TODO
 **/
public class Test {
    public static void main(String[] args) {
        //3.创建一个实现Callable接口的实现类的对象
        CallableNew callable =new CallableNew();
        //4.将Callable接口的实现类的对象作为参数传递到FutureTask构造器中,创建FutureTask的对象
        FutureTask futureTask = new FutureTask(callable);
        //5.将FutureTask类的对象作为参数传递到Thread类的构造器中,并且调用`start()`方法
        Thread thread=new Thread(futureTask);
        thread.start();

        try {
            //6.(可有可无)获取Callable中`call()`方法的返回值:`FutureTask.get()`
            Object sum=futureTask.get();
            System.out.println("总和为："+sum);
        } catch (InterruptedException e) {
            e.printStackTrace();
        } catch (ExecutionException e) {
            e.printStackTrace();
        }
    }

}
//1.创建一个实现Callable接口的实现类
class CallableNew implements Callable {
    //2.实现`call()`方法,线程任务声明在`call()`方法里面
    @Override
    public Object call() throws Exception {
        int sum=0;
        for(int i=0;i<100;i++){
            if(i%2==0){
                sum+=i;
                System.out.println(i);
            }
        }
        return sum;
    }
}

Callable接口与Runnable接口的比较

  1. call()可以有返回值
  2. call()可以抛出异常,被外面的动作捕获,获取异常的信息
  3. Callable支持泛型

使用线程池

线程池的由来

背景

经常创建和销毁,使用量特别大的资源,比如并发情况下的线程,对性能的影响比较大

思路

提前创建好多个线程,放入线程池,使用时直接获取,使用完放回池中,可以避免频繁的创建销毁,重复利用

好处

  1. 提高响应速度(减少创建线程的事件).
  2. 降低资源消耗(重复利用线程池中线程,不需要每次都创建).
  3. 便于线程管理.
    1. corePoolSize:核心池的大小.
    2. maximumPoolSize:最大线程数.
    3. keepAliveTime:线程没有任务时最多保持多长时间后会终止.

线程池相关API

ExecutorService🥸—-接口

  • 真正的线程池接口,常见之类ThreadPoolExecutor
    • void execute(Runnable command):执行任务/命令,没有返回值,一般用来执行Runnable.
    • <T>Future<T>submit(Callable<T>task):执行任务,有返回值,一般用来执行Callable

Executors😁—–工具类

  • 工具类,线程池的工厂类,用于创建并返回不同类型的线程池
    • Executors.newCashedThreadPool(): 创建一个可根据需要创建新线程的线程池
    • Executors.newFixedThreadPool(n):创建一个可重用固定线程数的线程池*
    • Executors.newSingleThreadExecutor()：创建一个只有一个线程的线程池
    • Executors.newScheduledThreadPol（n）：创建一个线程池，它可安排在给定延迟后运行命令或者定期地执行。

创建线程池的7个参数

  1. corePoolSize ： 线程池的核心线程数
  2. maximumPoolSize : 能容纳的最大线程数
  3. keepAliveTime : 空闲线程存活时间
  4. unit ： 存活的时间单位
  5. workQueue ： 存放提交但未执行任务的队列
  6. threadFactory ： 创建线程的工厂类
  7. handler ： 等待队列满后的拒绝策略

阿里巴巴java代码规范中对线程池的强制要求

要求

项目中创建多线程时，使用常见的三种线程池创建方式，单一、可变、定长都有一定问题，原因是FixedThreadPool和SingleThreadExecutor底层都是用LinkedBlockingQueue实现的，这个队列最大长度为Integer.MAX_VALUE，容易导致OOM。所以实际生产一般自己通过ThreadPoolExecutor的7个参数，自定义线程池：

代码实现

import com.google.common.util.concurrent.ThreadFactoryBuilder;
import java.util.concurrent.LinkedBlockingQueue;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.ThreadFactory;
import java.util.concurrent.TimeUnit;
//引入依赖包，创建线程池
private ThreadFactory namedThreadFactory = new ThreadFactoryBuilder().setNameFormat("thread-call-runner-%d").build();
private ExecutorService taskExe = new ThreadPoolExecutor(10,20,200L,TimeUnit.MILLISECONDS,new LinkedBlockingQueue<Runnable>(),namedThreadFactory);

创建线程池

创建线程池流程

  1. 创建线程池
  2. 创建实现Runnable/Callable接口的实现类
  3. 将Callable\Runnable实现类的对象作为参数调给submit()\execute()方法

代码实现

package com.lgx.threadthree;

import java.util.concurrent.Callable;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.ThreadPoolExecutor;

/**
 * @ClassName ThreadPool
 * @Author mrl
 * @Date 2021/12/11 11:13
 * @Version 1.6
 * @Description TODO
 **/

//2.创建实现Runnable接口的实现类
class RunnbleTestCase implements Runnable {

    @Override
    public void run() {
        for (int i = 0; i < 100; i++) {
            if (i % 2 == 0) {
                System.out.println(Thread.currentThread().getName()+":"+i);
            }
        }
    }
}
//2.创建实现Callable接口的实现类
class CallableTestCase implements Callable {

    @Override
    public Object call() throws Exception {
        for(int i=0;i<100;i++){
            if(i%2==0){
                System.out.println(Thread.currentThread().getName()+":"+i);
            }
        }
        return null;
    }

}

public class ThreadPool {
    public static void main(String[] args) {
        //ExecutorService只是一个接口，而实际上service所属的真正类是ThreadPoolExecutor
        //1.创建线程池
        ExecutorService service = Executors.newFixedThreadPool(20);
	
        ThreadPoolExecutor service1 = (ThreadPoolExecutor) service;//向下转型
        service1.setCorePoolSize(200);//设置核心线程的数量
        System.out.println(service.getClass());//获取该对象的所在类
	//3.将Runnable实现类的对象作为参数调给excute()方法
        service.execute(new RunnbleTestCase());  //适用于Runnable
       //3.将Callable实现类的对象作为参数调给submit()方法 
        service.submit(new CallableTestCase());//使用于Callable
    }

}
